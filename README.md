# 大航海时代任务向导

这是一个本地网页工具。日服任务可以直接输入日文，例如 `ワインをロンドンへ届ける`、`海賊討伐`、`神秘的な地図`。工具会先匹配本地任务库；没有命中时，会按交易、冒险、战斗、主线模板生成一套临时步骤。

## 打开方式

推荐双击 `start-web.cmd` 启动。它会打开本地网页，并允许网页直接按日服任务名查询 GVDB。

如果只是直接打开 `index.html`，本地任务库仍然可用，但不能自动访问 GVDB。

## 日服任务怎么导入

真正可用的关键是任务库。工具不会直接读取游戏客户端，也不建议抓取客户端内部数据；最稳妥的方式是把你看到的任务、攻略站资料、自己的笔记整理成表格，再导入工具。

推荐流程：

1. 打开 `data/jp-quest-template.csv`。
2. 每一行填一个日服任务。
3. 保存为 UTF-8 编码的 `.csv` 或 `.tsv` 文件。
4. 在网页右上角点导入按钮，选择这个文件。
5. 导入后直接输入日文任务名，就会输出对应步骤。

最少需要这些列：

- `クエスト名`：日文任务名，最重要。
- `分類`：任务类型，可填 `納品`、`交易`、`冒険`、`発見`、`海事`、`討伐`、`メイン`。
- `目的地`：交付城市、发现地点、战斗海域等。
- `手順`：详细步骤。

多值字段用 `|` 分隔，例如：

```csv
必要条件,準備,手順
ワイン必要数|ロンドン入港許可,船倉を空ける|ワインを購入する,依頼仲介人で受注する|ロンドンへ移動する|指定 NPC に納品する
```

## 支持的表头

CSV/TSV 可以用日文、中文或英文表头。常用映射如下：

| 字段 | 可用表头示例 |
| --- | --- |
| 任务名 | `クエスト名`、`依頼名`、`任务名`、`title` |
| 类型 | `分類`、`種別`、`类型`、`type` |
| 来源 | `資料`、`出典`、`版本`、`version` |
| 别名 | `別名`、`略称`、`别名`、`aliases` |
| 标签 | `タグ`、`标签`、`tags` |
| 接取地点 | `受注場所`、`受注`、`接取`、`start` |
| 目的地 | `目的地`、`納品先`、`発見場所`、`海域`、`destination` |
| NPC | `NPC`、`依頼人`、`報告先`、`納品NPC` |
| 条件 | `必要条件`、`必要スキル`、`前置`、`requirements` |
| 准备 | `準備`、`持ち物`、`准备`、`prep` |
| 步骤 | `手順`、`攻略`、`流れ`、`步骤`、`steps` |
| 备注 | `備考`、`メモ`、`注意`、`notes` |

## JSON 任务库格式

导入按钮也支持 JSON，顶层必须是数组：

```json
[
  {
    "title": "日文任务名",
    "type": "trade",
    "version": "日服 / 资料来源",
    "aliases": ["别名", "略称"],
    "tags": ["交易", "納品"],
    "start": "受注場所",
    "destination": "目的地",
    "npc": "報告先 NPC",
    "difficulty": "低 / 中 / 高",
    "estimatedTime": "预计耗时",
    "reward": "報酬",
    "requirements": ["必要条件"],
    "prep": ["准备清单"],
    "steps": ["详细步骤"],
    "notes": ["注意事项"]
  }
]
```

`type` 可选值：

- `trade`：交易、运输、納品、配達
- `adventure`：冒险、発見、探索、調査
- `battle`：海事、戦闘、討伐、護衛
- `story`：主线、メイン、ストーリー、シナリオ
- `generic`：暂未分类

导入后数据会保存在当前浏览器里。需要备份时点右上角导出按钮。

## 从巴哈姆特攻略百科导入

可以导入这个资料站：

`https://acg.gamer.com.tw/wikimenu.php?s=7501`

项目里已经加了本地导入器：

- `tools/import-bahamut.mjs`
- `import-bahamut.cmd`

最简单用法：

1. 先打开 `data/jp-aliases-template.csv`。
2. 把你知道的日服任务名填到 `日服任務名` 这一列。
3. 双击 `import-bahamut.cmd`。
4. 先选 `1` 跑 20 条测试。
5. 测试成功后，再按需要选 `2` 商人、`3` 冒险、`4` 海事、`5` 转职，或 `6` 全部。
6. 等它生成 `data/bahamut-import.json`。
7. 打开网页工具，点右上角导入按钮，选择 `data/bahamut-import.json`。

如果双击 CSV 出现乱码，优先打开 `data/jp-aliases-template.xlsx`。这个 Excel 版不会有编码问题。填好后另存为 `CSV UTF-8`，再运行导入器。

如果双击 `import-bahamut.cmd` 看起来没反应：

- 新版本会先显示菜单，不会直接开始抓整站。
- 选 `1` 是最快的测试模式，会生成 `data/bahamut-import-test.json`。
- 选 `6` 是整站详细导入，会很慢，看起来像跑很久是正常的。
- 窗口里会显示正在读取哪个分类，以及已经读取了多少任务详情。
- 如果失败，窗口不会立刻关闭；把窗口里的错误文字复制出来即可排查。

为什么需要 `jp-aliases-template.csv`：

巴哈资料是繁中任务名，日服游戏显示的是日文任务名。导入器可以抓到任务流程、地点、技能、报酬等资料，但如果没有日文别名，你输入日文任务名时不一定能命中。别名表就是用来把“巴哈任务名”和“日服任务名”连起来。

别名表示例：

```csv
巴哈任務名,日服任務名,備註
乳酪交貨,这里填日服实际显示的任务名,不确定就先留空
```

例如你只知道日服任务名 `商人への転職`：

1. 这条在巴哈对应的任务名是 `商人公會的困境`。
2. 在别名表里填：`商人公會的困境,商人への転職`。
3. 双击 `import-bahamut.cmd`，选择 `5` 导入转职任务。
4. 在网页工具里导入生成的 `data/bahamut-import.json`。
5. 之后搜索 `商人への転職` 就会命中这条任务。

只导入某个分类页时，可以运行：

```powershell
node tools/import-bahamut.mjs "https://wiki2.gamer.com.tw/wiki.php?n=7501:%E5%8C%97%E6%AD%90%E5%95%86%E4%BA%BA%E4%BB%BB%E5%8B%99&ss=7501&f=M" --details --out=data/bahamut-import.json --alias=data/jp-aliases-template.csv
```

导入整站目录时会比较慢，因为它会逐个打开任务详情页：

```powershell
node tools/import-bahamut.mjs "https://acg.gamer.com.tw/wikimenu.php?s=7501" --details --delay=700 --out=data/bahamut-import.json --alias=data/jp-aliases-template.csv
```

说明：

- `--details` 会读取每个任务详情页，才能拿到真实任务流程。
- `--delay=700` 是请求间隔，避免过快访问资料站。
- `--alias` 会合并日服任务名别名。
- 如果只想先测试少量任务，可以加 `--limit=20`。

## 按日服任务名从 GVDB 导入

如果你只知道日服任务名，优先用这个方式。最简单流程：

1. 双击 `start-web.cmd`。
2. 在网页输入日服任务名，例如 `商人への転職`。
3. 点“生成步骤”。
4. 如果本地库没有命中，网页会自动查询 GVDB。
5. 查到后会直接显示中文步骤和航行路线图，并存入本地任务库。

它会搜索日文资料站 GVDB：

`https://gvdb.mydns.jp/db/module/QuestDB/action/QuestSearch`

也可以用手动导入方式：

1. 双击 `import-japanese-quest.cmd`。
2. 输入日服任务名，例如 `商人への転職`。
3. 等它生成 `data/gvdb-import.json`。
4. 打开网页工具，点右上角导入按钮，选择 `data/gvdb-import.json`。
5. 之后直接在工具里输入日服任务名。

这个方式不需要填写 `jp-aliases-template.xlsx`，因为数据源本身就是日文任务名。

目前中文输出是规则转换：城市、NPC、常见物品、报酬、步骤会尽量转成中文；遇到无法可靠翻译的句子，会保留日文原文在备注里，避免误翻。

港口名已经补了 GVDB 首页“都市・港一覧”里的大部分常见港口，包含欧洲、非洲、印度、东南亚、东亚、美洲等区域。个别没有固定中文译名的港口会先用音译中文显示。

如果任务说明或备注里写了交付物，例如 `届け先 / 配達品`、`豚肉はオポルト／ポーツマスで入手可`、`マスタードを100樽、ここに持ってきて納品すること`，导入器会自动识别交付物、数量和交付地点。如果资料页没有写采购港，导入器会再去 GVDB 的物品页查这个交易品在哪里出售，并把推荐采购港插进路线里。

## 交易查询

网页右侧有“交易查询”面板：

1. 选“查物品”，输入 `猪肉`、`豚肉`、`ワイン` 这类物品名，可以查它在哪些港口出售。
2. 选“查港口”，输入 `奥波多`、`オポルト` 这类港口名，可以查这个港口交易所、道具屋、工房、造船等出售内容。
3. 查询会使用 GVDB 的交易 DB，结果会缓存到 `data/market-cache.json`，下次同样查询会更快。

中文别名目前覆盖常用物品和常见港口；如果中文名查不到，直接输入日文名通常更准。

如果浏览器里已经保存过旧版的不完整 GVDB 结果，重新用 `start-web.cmd` 打开后再搜同一个日服任务名，网页会自动尝试刷新一次。

以 `商人への転職` 为例，生成路线会包含：

1. 萨格雷斯接任务。
2. 前往里斯本，与商人公会会长对话。
3. 前往奥波多购买猪肉。
4. 前往伦敦，向交易所店主交付猪肉。
5. 交付后如果任务日志未完成，再回接任务城市报告。

网页结果页会根据这些步骤自动生成“航行路线图”。有网络时会使用 Leaflet + OpenStreetMap 精细底图，支持拖拽、放大、缩小、图层开关、港口悬停提示和当前任务路线高亮；普通港口点可以点击查看中文名和区域。若地图服务或网络不可用，会自动回退到本地 SVG 海图；未知港口仍会显示为节点路线。

路线不会再简单地用港口点对点直线相连。工具会优先使用“近岸航道”：先从港口出到外海参照点，再沿海岸线、海峡和海湾逐段前进，最后从目标港外海靠岸。常见航路会插入更密的参照点，例如葡萄牙外海、圣文森特角、直布罗陀海峡、阿尔沃兰海、比斯开湾、布列塔尼外海、英吉利海峡、好望角、莫桑比克海峡、亚丁湾、马六甲海峡等，并在步骤第一条显示“推荐航行路线”。如果采购港或目的港尚未发现，建议抵达最后一个航路参照点后沿海岸线低速搜索港口光点。

地图底图来自 OpenStreetMap 标准瓦片，页面右下角会保留必要署名。这个工具只做交互浏览，不会批量下载或离线缓存地图瓦片。

## 实时船位

网页现在可以在地图上显示“当前船位”。最简单的方式是在航行路线图下方手动输入纬度、经度，点“显示船位”，地图上会出现蓝色船位标记。

如果之后用外部工具读取到坐标，也可以把坐标推送给本地服务：

```powershell
Invoke-RestMethod -Uri "http://127.0.0.1:8765/api/position" -Method Post -ContentType "application/json; charset=utf-8" -Body '{"lat":36.75,"lon":3.06,"label":"当前船位","source":"ocr"}'
```

网页会每 2 秒读取一次 `/api/position`，收到新坐标后自动更新船位。清除船位可以在网页里点“清除”，或调用：

```powershell
Invoke-RestMethod -Uri "http://127.0.0.1:8765/api/position" -Method Delete
```

建议的安全实时方案是“屏幕 OCR”：只读取游戏画面上已经显示的坐标或测量信息，再把坐标发给网页。不要读取游戏内存、封包或客户端内部数据，这类方式容易违反游戏规则，也不适合放进这个工具。

以 `チーズの納品` 为例，会生成：

1. 伦敦接任务。
2. 去普利茅斯买奶酪。
3. 去加来找酒馆老板交付。
4. 备用采购港：安特卫普。

以 `マスタードの配送` 为例，会生成：

1. 萨格雷斯接任务。
2. 去阿尔及尔购买芥末 100 个。
3. 回萨格雷斯找任务委托人交付。
4. 备用采购港：科钦、第乌。

命令行用法：

```powershell
node tools/import-gvdb.mjs --query "商人への転職" --out=data/gvdb-import.json
```

### 游戏内坐标说明

游戏画面里显示的 `16058,3317` 这类数字不是经纬度，而是游戏自己的地图坐标。网页里的“实时船位”现在可以直接填写“游戏 X / 游戏 Y”，工具会根据 GVDB 港口坐标样本把它估算到可缩放地图上。

外部 OCR 工具也可以直接推送游戏坐标：

```powershell
Invoke-RestMethod -Uri "http://127.0.0.1:8765/api/position" -Method Post -ContentType "application/json; charset=utf-8" -Body '{"gameX":16058,"gameY":3317,"label":"当前船位","source":"ocr"}'
```

这个换算是用已知港口坐标做局部校准，不是官方投影公式；离已校准港口越近越准。后续如果补入更多港口的游戏坐标，船位会更贴近游戏里的真实位置。

### 地图校准

如果航行到远处后船位偏差变大，可以在网页里添加校准点：

1. 先让 OCR 或手动输入显示一次当前船位，确保网页里有 `游戏 X / 游戏 Y`。
2. 把船停在一个已知港口附近，或确认当前所在港口。
3. 在“地图校准”的“所在港口”里输入港口名，例如 `伦敦`、`ロンドン`、`里斯本`。
4. 点“添加校准”。

校准点会保存在当前浏览器里。一个校准点只能可靠修正这个港口附近，离它越远，工具会自动减弱这个校准点的影响，避免把远处位置带偏。两点可以改善一段航线附近，三个以上分布在同一海域的校准点会更适合做区域校准。可以在不同海域多加几个校准点；如果校准错了，点“清空校准”后重新添加。

### 屏幕 OCR 船位

最简单的执行方式：

1. 先双击 `start-web.cmd` 打开网页工具。
2. 让游戏窗口保持打开，并把游戏内坐标显示出来。
3. 双击 `start-ocr-position.cmd`。
4. 窗口会先停止旧 OCR，再让你选择：输入 `1` 重新拖拽框选坐标区域并使用手动扫描；输入 `2` 使用上次保存区域并手动扫描；输入 `3` 使用慢速自动扫描；输入 `4` 使用高精度模式。
5. 选择 `1` 后会出现一个半透明遮罩。按住鼠标左键拖住坐标数字区域，松开鼠标后会弹出放大预览；确认预览里能看见完整坐标后再点 `Use this area`。按 `ESC` 可以取消。
5. 之后 OCR 会每 2 秒识别一次，把 `游戏 X / 游戏 Y` 自动发送到网页地图。

框选时要包含整串数字，例如 `16058,3317`，但不要把太多小地图、海岸线或图标框进去。区域太窄时工具会要求重选；识别失败或数字经常跳动时，关闭 OCR 窗口，重新运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\live-position-ocr.ps1 -ResetRegion
```

OCR 使用本机的 Tesseract，只截取你指定的小区域，并推送到本机 `http://127.0.0.1:8765/api/position`。最后一次处理后的图片会保存在 `data/ocr-last.png`，方便检查是不是框到了正确位置。
为了提高小字体识别率，脚本会先把框选区域里的白色数字自动裁出来，再做放大识别；还会把逗号被误读成数字的情况按游戏坐标格式纠正，例如把 `24782234` 解析成 `247,2234`。窗口里显示的 `votes` 越高，说明结果越稳定。如果 `votes` 经常只有 1，建议重新框选一个更大的坐标显示区域，或把游戏窗口 / UI 缩放稍微调大。

OCR 默认开启稳定模式：窗口输出 `OK` 时才会发送到网页；输出 `WAIT` 时表示这一帧疑似误读或跳太远，暂时不更新地图。这样个别错字不会让船位乱跳。进入港口后如果游戏画面不再显示坐标，OCR 会先快速判断“当前没有坐标数字”，显示一次 `PAUSE coordinate hidden`，保留地图上的最后船位，出港后看到坐标会自动恢复。如果确实想关闭稳定过滤，可以手动运行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\live-position-ocr.ps1 -NoStable
```

OCR 现在默认推荐手动扫描模式：启动时选 `1` 或 `2` 后，程序平时不会定时截屏和识别，只有你在 OCR 窗口按 `S` 或空格时才扫描一次。这样最不容易影响游戏流畅度。启动时选 `3` 是慢速自动扫描，每 8 秒识别一次；选 `4` 会开启高精度模式，识别会更积极，但更容易让游戏不流畅。

如果 OCR 窗口显示 `WAIT  16058,3317 ... press A=accept M=correct`，但你看游戏画面确认这个坐标是对的，直接在 OCR 窗口按 `A`。工具会立刻把这次坐标发送到网页地图，同时把这张坐标截图加入训练。如果 OCR 读出来的数字不对，但截图区域是清楚的，按 `M`，手动输入正确坐标，例如 `16058,3317`，工具会用你输入的正确值更新地图并学习这一帧。

运行中如果发现框选不合适，直接在 OCR 窗口按 `R`，会重新进入拖拽框选，不需要关闭窗口。如果发现某次错误坐标已经影响后续判断，按 `C` 可以清空稳定坐标状态，工具会重新等待连续 OCR 读数确认后再发送。

如果想提高小字体精度，可以训练游戏字体模板：运行中按 `T`，确认预览里是当前坐标，然后在窗口里输入这行坐标的真实值，例如 `16058,3317`。多训练几组包含不同数字的坐标后，工具会优先用模板识别；模板数量不足时会保持保守，继续以原 OCR 作为兜底。

训练时窗口会显示 `You typed: ...`，这就是你刚刚输入的内容；如果逗号输成中文逗号或格式不对，会显示 `Input rejected`。如果框选区域里多了小地图箭头、边框、海岸线或图标碎点，训练会自动尝试只保留数字块；成功时会显示 `Training alignment: ...`。学习成功会显示 `Template file: ...` 和 `Unique digits trained: x/10`。模板识别现在默认关闭，因为当前小字体下原始 OCR 更稳；如果要临时测试模板识别，在 OCR 窗口按 `V` 开关。如果学错了，在 OCR 窗口按 `X` 可以清空所有数字模板；如果 OCR 窗口没有打开，可以双击 `clear-ocr-learning.cmd` 清空学习。

如果 OCR 窗口卡住或你想彻底停止识别，可以双击 `stop-ocr-position.cmd`。
